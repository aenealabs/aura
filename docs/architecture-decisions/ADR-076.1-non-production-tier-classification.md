# ADR-076.1: Non-Production Tier Classification for SBOM Attestation

## Status

Accepted (deploy phases gated by ADR-076 Phase 2 / issue #195 P2)

## Date

2026-05-15

## Relationship to ADR-076

This is an addendum to [ADR-076: SBOM Attestation and Supply Chain Security](ADR-076-sbom-attestation-supply-chain.md). It does not change ADR-076's pillars; it clarifies one classification axis that was ambiguous when ADR-076 was written: **the attestation chain says "we built and signed this," not "this is production-grade." Test / dev images that get built and pushed through the same pipeline need an explicit tier label so downstream auditors don't conflate them with production-tier artifacts.**

## Context

ADR-076 deployed CycloneDX + SPDX SBOMs, Sigstore signatures, and Rekor attestations against the platform's production container images. The pipeline was scoped to "production images" with the implicit assumption that everything routed through it was production-tier.

Issue #195 (2026-05-15 ultrareview follow-up) proposed promoting the local-developer Linux test harness (`deploy/docker/test-harness/`) from a per-dev local build to a CI-built, ECR-distributed shared image. The harness:

1. Is **not** production-grade -- it pulls a per-developer test stack including torch with CUDA wheels, build-essential, libcairo, hdbscan with native compilation. The attack surface is intentionally broader than any production image.
2. Is **never deployed to production** -- it runs pytest on a Linux runtime to validate code before merge.
3. **Should** still pick up the SBOM + Sigstore pipeline -- developer-laptop blast radius for a compromised wheel is real (see Sally's 2026-05-14 supply-chain review on the original PR), and routing the harness through the same controls is cheaper than building parallel infrastructure.

The risk if we route it through ADR-076's pipeline without a tier marker: an auditor querying the Rekor transparency log for "Aenea Labs production attestations" sees the harness image alongside the production cluster and can't tell which is which. That dilutes the value of the production attestation -- the strongest claim "all our production code is signed" weakens to "all our signed code includes our production code somewhere in it."

## Decision

Introduce an explicit two-axis classification for every image that flows through ADR-076's attestation chain:

| Axis | Values | Where it's set |
|------|--------|----------------|
| **Tier** | `production` \| `non-production` | OCI label `tier=<value>` + Sigstore predicate field `tier` |
| **Scope** | `runtime` \| `test-harness` \| `tooling` \| `base-image` | OCI label `scope=<value>` + Sigstore predicate field `scope` |

### Routing Rules

1. **Production images** (`tier=production`) ship to their existing per-service ECR repository (e.g. `aura-api`, `aura-orchestrator`). No change from ADR-076.

2. **Non-production images** (`tier=non-production`) ship to a **distinct ECR repository** whose name carries the non-production marker:
   - Test harness: `aura-test-harness` (issue #195 P2)
   - Future tooling images: `aura-tooling-<name>`
   - **Never** as a tag under a production repo. Repository separation is the load-bearing audit boundary; the tier label is a defense-in-depth confirmation.

3. **Sigstore predicates** for non-production images include a top-level field:
   ```json
   {
     "_type": "https://in-toto.io/Statement/v0.1",
     "predicateType": "https://aenealabs.com/attestations/v1",
     "predicate": {
       "tier": "non-production",
       "scope": "test-harness",
       "purpose": "Local-developer pytest harness; not deployed.",
       "...": "..."
     }
   }
   ```
   Auditors filter on `predicate.tier == "production"` to scope a query to production-only attestations.

4. **CodeBuild assertions:** the buildspec that builds a non-production image must assert that its target repo URL matches `aura-test-harness` or `aura-tooling-*`, fail-fast on production-repo accidents.

### Why an addendum, not a new ADR

The pillars in ADR-076 (CycloneDX, SPDX, Sigstore, Rekor, dependency-confusion detection, license compliance, VEX) all stay -- we're not replacing or extending them. We're disambiguating one classification dimension that turned out to be load-bearing for audit. A full new ADR would be heavier than this disambiguation deserves; an addendum keeps the cross-reference tight.

## Consequences

### Positive

- Auditors can query "all attestations for production-tier images" cleanly.
- Test/tooling images can adopt the SBOM/Sigstore pipeline without diluting production attestations.
- Repository separation is the load-bearing boundary; the tier label is a verifiable cross-check.
- Issue #195's "wire the harness into the same pipeline" goal lands without compromising ADR-076's audit posture.

### Negative

- Adds one new OCI label + one new predicate field; downstream tooling that filters attestations needs a small update if it didn't already filter on repo name.
- Requires the CodeBuild buildspec for each non-production image to set the labels at build time. The existing production buildspec patterns don't have this; needs to be added.

### Neutral

- The tier classification is independent of severity / hardening level. A `tier=non-production` image is not necessarily less secure -- the harness uses the same hardened base, scanned wheels, and Sigstore signing. The label just records "this isn't slated for production runtime use."

## Implementation Notes

The OCI labels are set in the Dockerfile (image-time, immutable):

```dockerfile
LABEL tier="non-production" \
      scope="test-harness" \
      org.opencontainers.image.title="Aura Test Harness"
```

The Sigstore predicate fields are set by the signing step in the CodeBuild buildspec (build-time, signed):

```bash
cosign attest \
  --predicate predicate.json \
  --type 'https://aenealabs.com/attestations/v1' \
  "${ECR_URI}:${TAG}"
```

where `predicate.json` carries `tier`, `scope`, and `purpose`.
