7. Tool System & MCP Integration

Agents perform tasks by invoking tools through the Model Context Protocol (MCP). Aura does not use a "skills" system — instead, tools are classified into 4 risk tiers with runtime governance, and presented to agents via a hierarchical registry (Manus pattern) that caps visible tools at 20–30 to prevent context confusion.

Tools, Not Skills Every agent action — from reading a graph to deploying to production — is modeled as a tool invocation passing through capability enforcement middleware. Tools are executable operations with explicit risk classification, rate limits, and audit requirements.

MCP Tool Server (Internal)

src/services/mcp_tool_server.py

Exposes Aura's internal services as MCP tools. Five handler categories serve all internal agent operations.

Hierarchical Tool Registry (Manus Pattern)

src/services/hierarchical_tool_registry.py

Context-efficient tool presentation. ~20 atomic tools always available; domain-specific tools loaded on demand; expert tools for complex scenarios only. Research-backed cap of 20–30 tools to prevent hallucinated parameters and wrong tool calls.

8. Capability Governance (ADR-066)

Every tool invocation passes through a 4-layer enforcement stack that evaluates agent identity, tool risk tier, execution context, and dynamic grants before deciding ALLOW, DENY, ESCALATE, or AUDIT_ONLY.

Tool Invocation Flow
  Agent Code
      │
      ▼
  invoke_tool("tool_name", {params})
      │
      ▼
  ┌─────────────────────────────────────────────────────────────┐
  │  CapabilityEnforcementMiddleware                            │
  │  ├─ Extract: agent_id, agent_type, context                 │
  │  ├─ Resolve: base policy + overrides + dynamic grants      │
  │  ├─ Evaluate: policy.can_invoke(tool, action, context)     │
  │  └─ Decision:                                              │
  │       ALLOW ──────► Execute tool                           │
  │       DENY ───────► Return error                           │
  │       ESCALATE ───► Queue for HITL approval                │
  │       AUDIT_ONLY ─► Execute + flag for review              │
  └─────────────────────────────────────────────────────────────┘
      │
      ├─► Internal tools ──► MCP Tool Server
      ├─► External tools ──► MCP Gateway Client
      └─► Native agents ──► Adapter (Enterprise mode)
      │
      ▼
  Audit Service ──► DynamoDB + CloudWatch

4-Tier Tool Classification

TierRisk LevelDefaultAuditExample Tools
Level 1 SAFEALLOW10% sample semantic_search, describe_tool, get_sandbox_status, list_tools
Level 2 MONITORINGALLOW100% query_code_graph, get_code_dependencies, query_embeddings, query_audit_logs
Level 3 DANGEROUSDENY100% index_code_embedding, destroy_sandbox, write_config, commit_changes
Level 4 CRITICALDENY + HITL100% provision_sandbox, deploy_to_production, rotate_credentials, modify_iam_policy

Per-Agent Policies

src/services/capability_governance/policy.py

Each agent type gets an AgentCapabilityPolicy specifying allowed/denied tools, permitted contexts (test, sandbox, production), rate limits, and whether the agent can elevate child permissions. Evaluation order: explicit deny → context restriction → explicit allow → tier defaults → default decision.

Dynamic Capability Grants

src/services/capability_governance/dynamic_grants.py

Temporary elevation through HITL approval. Four grant scopes: SINGLE_USE (1 hr), SESSION (8 hr), TASK_TREE (24 hr), TIME_BOUNDED (custom). Stored in DynamoDB with TTL expiration.
