1 Step

Define Your Autonomy Levels

The foundation of any HITL system is a configurable autonomy spectrum. Rather than building a binary "always-approve" gate, define discrete levels that map to your organization's risk tolerance, regulatory posture, and operational maturity. This lets you tune oversight per-team, per-environment, or per-repository without rewriting control logic.

Level 1

FULL_HITL

Human approval required for all operations regardless of severity. Every AI-generated patch, recommendation, or automated action is held until a designated reviewer explicitly approves it.

Best for: Defense contractors, healthcare systems, government agencies. Regulatory frameworks like CMMC, ITAR, and HIPAA often mandate this level.

Level 2

CRITICAL_HITL

Human approval for HIGH and CRITICAL severity only. Medium and low severity operations proceed automatically with full audit logging. Balances velocity with oversight.

Best for: Financial services, enterprise SaaS, large engineering organizations with SOC 2 or SOX compliance requirements.

Level 3

AUDIT_ONLY

All operations auto-proceed, but every decision is logged with full context: what was detected, what action was taken, confidence score, and reasoning. Post-hoc review replaces pre-execution approval.

Best for: Internal tools, staging environments, low-risk repositories where speed matters more than pre-approval.

Level 4

FULL_AUTONOMOUS

Fully automated except for non-bypassable guardrails (Step 2). Minimal logging overhead. The system operates independently with guardrails as the only hard stops.

Best for: Development/test environments, startups with high risk tolerance, internal experimentation platforms.

2 Step

Implement Non-Bypassable Guardrails

Regardless of which autonomy level an organization selects, certain operations must always require human approval. These guardrails are checked before any policy evaluation occurs. They cannot be disabled by configuration, overrides, or even the master HITL toggle. This is the safety net beneath all other controls.

Production Deployments

Any change pushed to a production environment. Blast radius is maximum; rollback is costly. A human must verify the change is intentional and tested.

Credential / Secret Modifications

API keys, database passwords, service account tokens. Unauthorized changes here can compromise entire systems. Rotation must be deliberate.

Access Control / IAM Changes

Role assignments, permission grants, policy attachments. Privilege escalation is the top attack vector. Every IAM change needs a human eye.

Database Schema Migrations

Column drops, type changes, index modifications on production data. These operations are often irreversible and can cause data loss or extended downtime.

Infrastructure Modifications

Cloud resource provisioning, network configuration, security group changes. Misconfigured infrastructure can expose systems to the public internet.

Python guardrails.py 
# Non-bypassable guardrail operations.
# These are checked BEFORE any policy evaluation.
# Using frozenset to prevent runtime modification.

NON_BYPASSABLE_GUARDRAILS = frozenset({
    "production_deployment",
    "credential_modification",
    "access_control_change",
    "database_migration",
    "infrastructure_change",
})


def requires_approval(
    operation: str,
    severity: str,
    policy: AutonomyPolicy,
) -> bool:
    """
    Determine if an operation requires human approval.

    Guardrails are evaluated FIRST and cannot be bypassed
    by any policy configuration or override.
    """
    # Step 1: Check guardrails (highest priority)
    if operation in NON_BYPASSABLE_GUARDRAILS:
        return True  # Always requires approval

    # Step 2: Check master toggle
    if not policy.hitl_enabled:
        return False

    # Step 3: Evaluate policy hierarchy
    level = policy.get_autonomy_level(severity, operation)

    if level == AutonomyLevel.FULL_HITL:
        return True

    if level == AutonomyLevel.CRITICAL_HITL:
        return severity.upper() in ("HIGH", "CRITICAL")

    # AUDIT_ONLY and FULL_AUTONOMOUS: no approval needed
    return False
Why These Are Non-Negotiable These five operations share common characteristics that make automated execution unacceptable in any environment: Regulatory requirements mandate human authorization for production changes (SOX Section 404, PCI-DSS Requirement 6). Liability exposure means the organization, not the AI, bears legal responsibility for consequences. Irreversibility -- dropped database columns, rotated credentials, and deleted IAM roles cannot be undone with a rollback button. Blast radius -- each operation can affect an entire system, all users, or all downstream services simultaneously. Security boundary -- these operations cross trust boundaries where automated systems should not have unilateral authority.

4 Step

Design the Policy Resolution Hierarchy

When your system receives an operation request, it must resolve which autonomy level applies. With guardrails, repository overrides, operation overrides, severity overrides, and a default level all in play, you need a deterministic priority order. The first matching rule wins.

Resolution Priority (Highest to Lowest)
  Priority 1  GUARDRAILS                 Always enforced. Cannot be overridden.
     |        Is the operation in the guardrail set?
     |        YES --> FULL_HITL (stop evaluation)
     v
  Priority 2  REPOSITORY OVERRIDES       Per-repo policies for sensitive codebases.
     |        Does the target repo have an override?
     |        YES --> Use repo-specific level (stop)
     v
  Priority 3  OPERATION OVERRIDES        Per-operation-type policies.
     |        Does the operation type have an override?
     |        YES --> Use operation-specific level (stop)
     v
  Priority 4  SEVERITY OVERRIDES         Per-severity policies.
     |        Does the severity level have an override?
     |        YES --> Use severity-specific level (stop)
     v
  Priority 5  DEFAULT LEVEL              Fallback when no overrides match.
             Use the policy's default autonomy level.

Resolution Examples

Example A: Payment Gateway, MEDIUM Severity

Context: A MEDIUM-severity dependency update in the payment-gateway repository.

Policy: Enterprise Standard preset (default: CRITICAL_HITL, MEDIUM override: AUDIT_ONLY).

Resolution path:
1. Guardrails? No, this is a dependency_update operation.
2. Repository override for payment-gateway? Yes -- FULL_HITL.
3. Result: Requires human approval.

Even though the severity is only MEDIUM and the severity override would allow AUDIT_ONLY, the repository override takes priority because payment-gateway handles financial transactions.

Example B: Main App, LOW Severity

Context: A LOW-severity code style fix in the main-app repository.

Policy: Enterprise Standard preset (default: CRITICAL_HITL, LOW override: FULL_AUTONOMOUS).

Resolution path:
1. Guardrails? No, this is a code_style_fix operation.
2. Repository override for main-app? No override configured.
3. Operation override? No override for code_style_fix.
4. Severity override for LOW? Yes -- FULL_AUTONOMOUS.
5. Result: Auto-proceed, no approval required.

No guardrail, no repo override, no operation override. The severity override for LOW maps to FULL_AUTONOMOUS, so the system proceeds automatically.

Python policy_resolver.py 
def get_autonomy_level(
    self,
    severity: str,
    operation: str,
    repository: str = "",
) -> AutonomyLevel:
    """
    Resolve autonomy level using the 5-level priority hierarchy.
    First matching rule wins.
    """
    # Priority 1: Guardrails (always FULL_HITL)
    if operation in self.guardrails:
        return AutonomyLevel.FULL_HITL

    # Priority 2: Repository-specific override
    if repository and repository in self.repository_overrides:
        return self.repository_overrides[repository]

    # Priority 3: Operation-specific override
    if operation in self.operation_overrides:
        return self.operation_overrides[operation]

    # Priority 4: Severity-specific override
    if severity.upper() in self.severity_overrides:
        return self.severity_overrides[severity.upper()]

    # Priority 5: Default level
    return self.default_level
5 Step

Build the Approval Workflow Pipeline

The approval workflow is a 5-stage pipeline that takes an AI-generated action from detection through human review to deployment. Each stage has clear entry/exit criteria, and the system must handle both the happy path (approval) and rejection gracefully.

5-Stage Approval Pipeline
  STAGE 1              STAGE 2              STAGE 3              STAGE 4              STAGE 5
  Detection &          Sandbox              Approval             Human               Deployment /
  Generation           Validation           Request              Review              Closure

  +-------------+      +-------------+      +-------------+      +-------------+      +-------------+
  |             |      |             |      |             |      |             |      |             |
  | AI detects  | ---> | Isolated    | ---> | Create HITL | ---> | Reviewer    | ---> | Deploy with |
  | vuln and    |      | environment |      | request     |      | examines    |      | 24hr        |
  | generates   |      | tests the   |      | + notify    |      | evidence    |      | monitoring  |
  | patch       |      | patch       |      | reviewers   |      |             |      |             |
  |             |      |             |      |             |      |             |      |             |
  +-------------+      +-------------+      +-------------+      +------+------+      +-------------+
                                                                        |
                                                              +---------+---------+
                                                              |         |         |
                                                          APPROVE   REJECT   CHANGES
                                                              |         |         |
                                                          Deploy    Log reason  Return to
                                                          + monitor  + feedback  Stage 1
                                                          + audit    + cleanup   with notes

Stage 1: Detection and Generation

The AI agent scans the codebase, identifies a vulnerability (CVE match, SAST finding, dependency alert), and generates a remediation patch. The output includes the original vulnerable code, the proposed fix, a confidence score, and the reasoning chain.

Stage 2: Sandbox Validation

The patch is applied in an isolated environment (no network egress, ephemeral compute). Automated checks run: syntax validation, existing unit tests, security scan of the patch itself, and basic performance regression tests. Results are attached to the approval request.

Stage 3: Approval Request

The HITL service creates a structured approval request with all context. The notification service dispatches alerts to designated reviewers via configured channels: email, Slack, dashboard notification, or PagerDuty for CRITICAL severity.

Stage 4: Human Review

The reviewer sees a unified view: original code, AI patch (diff), sandbox results, confidence score, and reasoning. Three decisions are available: APPROVE (proceed to deployment), REJECT (log reason, feed back to AI), or REQUEST_CHANGES (return to Stage 1 with reviewer notes).

Stage 5: Deployment or Closure

If approved: the patch is deployed through the standard CI/CD pipeline with 24-hour enhanced monitoring. If rejected: the reason is logged, the AI receives the feedback for learning, and all sandbox resources are cleaned up.

Design Principle: Evidence-Based Review The approval request should contain everything a reviewer needs to make a decision without leaving the review interface. Critical Design Decision: Never Auto-Approve on Timeout A common anti-pattern is to auto-approve operations when the timeout expires. This defeats the purpose of HITL entirely. Instead, escalate high-severity items to backup reviewers and expire-and-requeue lower-severity items. The goal is to get a human decision, not to bypass the human decision.
7 Step

Build the Audit Trail

Every HITL decision must produce a compliance-grade audit record. Auditors need to answer three questions: who made the decision, what evidence did they see, and when did it happen. Your audit trail must be immutable, retained for regulatory periods, and linked to the specific policy configuration that was active at decision time.

Events to Log

Event Type Trigger Critical Fields
approval_request_created AI generates a patch that requires HITL review Vulnerability details, severity, assigned reviewer, policy snapshot
notification_sent Reviewer is notified of pending approval Channel (email, Slack, dashboard), recipient, delivery confirmation
human_decision Reviewer approves, rejects, or requests changes Decision, reviewer identity, reasoning, time spent reviewing
escalation_triggered Timeout threshold reached, escalation executed Original reviewer, escalation level, backup reviewer assigned
deployment_executed Approved patch deployed to target environment Commit SHA, environment, deployment method, monitoring config
request_expired Timeout reached without decision Total escalations attempted, re-queue status, final disposition

Immutable Hash Chain Each audit event includes a SHA-256 hash of its contents and a reference to the previous event's hash. This creates a tamper-evident chain similar to a blockchain ledger. If any event is modified after the fact, the hash chain breaks and the tampering is detectable. Store events in append-only storage (S3 with Object Lock, CloudWatch Logs with retention policies) and replicate to a separate AWS account for defense in depth.

AWS-Native Approach

Best for: Teams already on AWS with compliance requirements that benefit from managed services and native encryption.

Advantages: Managed scaling, built-in encryption at rest, native IAM integration, CloudTrail audit coverage, GovCloud availability.

Trade-offs: Vendor lock-in, higher cost at scale, limited customization of managed service behavior.

Open-Source / Multi-Cloud Approach

Best for: Teams that need portability, operate across clouds, or want full control over every component.

Advantages: No vendor lock-in, full customization, lower per-unit cost at high scale, community-driven improvements.

Trade-offs: Operational overhead for self-hosted services, must implement encryption and access control manually, no single-vendor support.

Architecture Recommendation: Start with Managed, Extract When Needed Begin with managed services (Step Functions + DynamoDB + SNS) to validate your workflow logic quickly. Once the approval patterns stabilize, evaluate whether the managed service costs or limitations justify migrating specific components to self-hosted alternatives. The policy resolution logic (Step 4) and guardrail enforcement (Step 2) are pure application code and remain portable regardless of infrastructure choice.
9 Step

Key Takeaways

Building a production-grade HITL system comes down to five principles. If you internalize these, the implementation details follow naturally.

1

HITL Is Configurable, Not Fixed

Do not hardcode a single approval mode. Build a spectrum of autonomy levels and let organizations choose where they sit based on their risk tolerance and regulatory requirements.

2

Guardrails Are Absolute

Certain operations must always require human approval, regardless of policy configuration. These guardrails are the non-negotiable safety net that protects against catastrophic automated actions.

3

Presets Simplify Compliance

Industry presets encode regulatory best practices into ready-to-use configurations. Teams get compliant faster and customize only what they need to diverge from the standard.

4

Every Decision Is Audited

The audit trail is not optional. Every approval, rejection, escalation, and auto-proceed must produce an immutable record with actor identity, evidence, timestamp, and policy context.

5

Escalation Prevents Bottlenecks

Without escalation, a single unresponsive reviewer blocks the entire remediation pipeline. Timeout-based escalation ensures that human oversight scales beyond any individual reviewer's availability.

From Blueprint to Production This runbook provides the architectural blueprint. The implementation details -- table schemas, API contracts, IAM policies, CloudFormation templates -- will vary based on your technology choices from Step 8. Start with the guardrails (Step 2) and policy resolution (Step 4) as your first sprint. These are pure application logic with zero infrastructure dependencies and can be tested with unit tests before you provision any cloud resources.
