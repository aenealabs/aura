# ADR-085.1: Object-Code Verification Track (DO-178C 6.4.4.2c)

**Status:** Proposed (Phase 1 scaffold shipped via issue #210)
**Date:** 2026-05-22
**Author:** Project Aura Team
**Parent ADR:** ADR-085 (Deterministic Verification Envelope)
**Related:** ADR-093 (Neptune-backed taint resolver -- source-side symbol set)

---

## Context

DO-178C 6.4.4.2c requires, for software at DAL A, that the verifier
demonstrate the compiled **object code** matches the **source code**
reviewed, tested, and structurally covered by the existing DVE
pillars. ADR-085 Phase 4 declares the policy hook
(`DALCoveragePolicy.requires_object_code_verification`) but did not
ship an implementation.

The Aura-vs-competitor comparison doc surfaced this as a documented
gap blocking any DAL A pilot. PR #213 (issue #209) closed the
"all-three-runs-are-Claude" gap with a non-LLM second voice; this
ADR closes the parallel "no object-code verification" gap.

The full implementation requires real compiler + disassembler
toolchain integration (gcc-arm / xtensa / objdump) plus symbol-table
comparison plus an embedded-target test harness -- a multi-week
effort. This ADR breaks the work into three sequenced phases so the
gate and policy hook ship now and the toolchain integration lands
incrementally.

---

## Decision

Introduce an **`ObjectCodeVerifierPort`** Protocol under
`src/services/verification_envelope/object_code/` that integrates
with the existing `DALCoveragePolicy.requires_object_code_verification`
flag through a new **`ObjectCodeGate`** service. Implementations
phase in over three deliverables:

| Phase | Deliverable | Issue | Status |
|---|---|---|---|
| 1 | Port + stub verifier + DAL gate + scoping doc | #210 | **Shipped** |
| 2 | `GccObjdumpVerifier` (x86_64 + ARM cross-compile) | TBD | Planned |
| 3 | `EmbeddedTargetVerifier` (ARM/PowerPC openocd/pyocd) | TBD | Planned |

### Phase 1 (this ADR / #210) -- scaffold

- `ObjectCodeVerifierPort` (Protocol): `verify(source, target_triple, toolchain, scratch_dir) -> ObjectCodeVerdict`
- `ObjectCodeVerdict` dataclass with `status ∈ {MATCHED, MISMATCHED, NOT_IMPLEMENTED, TOOLCHAIN_UNAVAILABLE}`, symbol counts, unexpected/missing symbol lists, rationale, latency.
- `NotImplementedObjectCodeVerifier` -- deterministic stub returning `NOT_IMPLEMENTED`. Wired by default into every DAL A profile.
- `ObjectCodeGate.evaluate(policy, source, ...)` -- consults `policy.requires_object_code_verification`, runs the verifier, returns `ObjectCodeGateResult` (`passed`, `required`, `verdict`, `rationale`). Only `MATCHED` passes; the stub status, `MISMATCHED`, and `TOOLCHAIN_UNAVAILABLE` all block COMPLETED at DAL A.
- **Fail-closed when no verifier is configured for a policy that requires it.**

### Phase 2 -- `GccObjdumpVerifier`

Targets: `x86_64-linux-gnu`, `arm-none-eabi`, `armv7-eabi`.

Algorithm:

1. Write source to scratch dir.
2. Compile via the configured toolchain (`gcc-13` / `arm-none-eabi-gcc-12`) with `-O0 -g -c -o out.o`.
3. Run `objdump --syms out.o` and parse the symbol table.
4. Source-side symbol set is supplied by the ADR-093 Phase 2 symbol extractor (already shipped).
5. Compare sets:
   - Symbols in object but not in source -> populate `unexpected_symbols`. Compiler-introduced symbols (e.g., `__cxa_atexit`, `_GLOBAL__sub_I_*`) get filtered against a maintained allowlist; anything not on the allowlist trips `MISMATCHED`.
   - Symbols in source but not in object -> populate `missing_symbols`. Dead-code elimination is a legitimate reason; the verifier emits a `MATCHED` verdict only when missing symbols are all `static` and provably uncalled (LLVM-style call-graph), otherwise `MISMATCHED`.

Required policy additions:

- `DALCoveragePolicy.object_code_target_triple: str` (default `"x86_64-linux-gnu"`).
- `DALCoveragePolicy.object_code_toolchain: str` (default `"gcc-13"`).
- `DALCoveragePolicy.object_code_compiler_symbol_allowlist: tuple[str, ...]`.

### Phase 3 -- `EmbeddedTargetVerifier`

For DAL A avionics workloads that ship on ARM Cortex-M or PowerPC
e500 cores, on-target test execution is the only acceptable evidence.

Architecture:

- Target board harness via `pyocd` (ARM) or `gdbserver` over JTAG (PowerPC).
- Source is cross-compiled with the appropriate toolchain (`arm-none-eabi-gcc`, `powerpc-eabi-gcc`).
- Image flashed to a hardware target in a sandboxed lab; test harness drives the unit tests over the debug probe.
- Structural-coverage instrumentation via `gcov`-on-target or a vendor tool (LDRA Testbed).
- Verdict: `MATCHED` only when (a) compile succeeded for the target triple, (b) on-target test pass-rate equals 100%, (c) structural coverage matches the host-side coverage report.

Phase 3 deliverables are explicitly out of scope for this ADR --
they require physical lab hardware + vendor licensing decisions that
will be costed separately.

## Threat model

| Threat | Mitigation |
|---|---|
| Compiler inserts hidden code path the source review didn't see | Symbol-set diff with a curated allowlist; unexpected symbols off the allowlist block `MATCHED`. |
| Compiler optimisation eliminates a tested path | `-O0` for verification builds; production-grade `-O2` build runs through a separate Phase 3 on-target validation. |
| Toolchain compromise / supply-chain attack on gcc | Toolchain pinned to a content-addressed image; ADR-088 supply-chain verifier covers the binary. |
| Operator runs the gate without configuring a verifier | Gate fails closed (`passed=False`) and emits `"no verifier is configured"` rationale. |
| Source-side symbol extractor has a bug | ADR-093 Phase 2 symbol extractor has its own 321-test suite; integration test (Phase 2 deliverable) will cross-check both sides. |

## Trade-offs

**Why a separate ADR rather than amending ADR-085?** ADR-085 is
already Accepted and shipped; reopening it adds review surface for
the unrelated Phases 1-5 that are stable. ADR-085.1 isolates the
object-code work so it can mature without churning the parent.

**Why ship a stub verifier rather than nothing?** The gate has to
fail closed for DAL A profiles regardless. A stub verifier with
`NOT_IMPLEMENTED` status documents the gap visibly in every audit
trail; the absence of a stub would route the same operator to a
"no verifier configured" message that looks like a misconfiguration
rather than a known scoped gap.

**Why not require Phase 2 to ship before Phase 1?** The gate hook
(`requires_object_code_verification=True`) already exists in
`DALCoveragePolicy`. Without the Phase 1 gate code, that flag is
dead: it advertises a check that never runs. Phase 1 makes the flag
load-bearing immediately even if the underlying implementation is
deferred.
