2. HITL Approval Service

The core approval engine manages the full lifecycle of approval requests from creation through decision, expiration, and escalation. Supports MOCK mode (in-memory) for testing and AWS mode (DynamoDB) for production.

HITLApprovalService

src/services/hitl_approval_service.py · 1,860 lines

Central approval service with DynamoDB persistence, partition-spread queries (10 status buckets), 24-hour timeout with escalation, and conditional writes to prevent race conditions.

Approval Lifecycle

StatusDescriptionTransitions To
PENDINGAwaiting reviewer decisionApproved, Rejected, Expired, Cancelled
APPROVEDReviewer approved the requestTerminal
REJECTEDReviewer rejected the requestTerminal
EXPIREDTimeout elapsed without decisionEscalated
ESCALATEDForwarded to backup reviewersApproved, Rejected, Expired
CANCELLEDProgrammatically cancelledTerminal

Timeout & Escalation

Default timeout: 24 hours. Warning sent at 75% (18 hrs). On expiration, request escalates to backup reviewers. Maximum 2 escalations before final expiration. TTL: 90 days in DynamoDB.

24h Default Timeout Max 2 Escalations

DynamoDB Schema

Primary key: approvalId. Three GSIs for efficient queries: StatusBucketIndex (scatter-gather across 10 buckets), PatchIdIndex, and ReviewedAtIndex.

3. Autonomy Framework (ADR-032)

Organizations select from 7 industry-tailored presets that map to 4 autonomy levels. A master HITL toggle and 5 non-bypassable guardrails provide safety rails even in fully-autonomous mode. Target: 85% autonomous operation for commercial enterprises.

4 Autonomy Levels

LevelHITL RequiredDescription
FULL_HITLAll actionsEvery agent action requires human approval
CRITICAL_HITLHIGH + CRITICAL severityOnly high-risk actions require approval
AUDIT_ONLYNone (post-hoc audit)Actions logged, reviewed after the fact
FULL_AUTONOMOUSNoneNo approval required (guardrails still apply)

7 Industry Presets

PresetDefault LevelTarget IndustryKey Overrides
defense_contractorFULL_HITLGovCloud, CMMC L3+Even LOW severity → CRITICAL_HITL
financial_servicesFULL_HITLSOX, PCI-DSSLOW → AUDIT_ONLY
healthcareFULL_HITLHIPAALOW → AUDIT_ONLY
fintech_startupCRITICAL_HITLGrowth-stage fintechLOW/MEDIUM → AUDIT_ONLY
enterprise_standardCRITICAL_HITLFortune 500LOW → AUTONOMOUS, MEDIUM → AUDIT
internal_toolsFULL_AUTONOMOUSInternal developmentprod_deploy → CRITICAL_HITL, cred_mod → FULL_HITL
fully_autonomousFULL_AUTONOMOUSDev/test environmentsMaster HITL toggle OFF

5 Non-Bypassable Guardrails

These always require HITL — even when the master toggle is OFF production_deployment · credential_modification · access_control_change · database_migration · infrastructure_change
Override Resolution Order (Highest Priority First)
  1. Guardrails ──────── Always enforced regardless of policy (non-bypassable)
  2. Operation overrides  Per-operation configs (e.g., prod_deploy → FULL_HITL)
  3. Repository overrides Per-repo configs (e.g., payments-service → CRITICAL_HITL)
  4. Severity overrides ─ Per-severity configs (e.g., LOW → AUDIT_ONLY)
  5. Default level ────── Policy default (e.g., CRITICAL_HITL)

4. Real-Time Agent Intervention (ADR-042)

Execution checkpoints pause agents mid-action for human approval via WebSocket streaming. Trust rules enable auto-approval for frequently-approved action patterns.

5. Dynamic Capability Grants (ADR-066)

When an agent requests a tool above its base permission level, the system escalates to HITL. On approval, a time-bounded grant is issued with precise scope and usage limits.

Grant ScopeDurationUse CaseMax Usage
SINGLE_USE60 minutesOne-time privileged operation1 invocation
SESSION8 hoursDuration of agent session100 invocations
TASK_TREE24 hoursAll agents in execution tree100 invocations
TIME_BOUNDED4 hours (default)Custom time window100 invocations

DynamicGrantManager

src/services/capability_governance/dynamic_grants.py

Creates, validates, tracks, and revokes grants. DynamoDB persistence with TTL auto-expiration. Max 10 active grants per agent. Usage counted per invocation.

6. Sandbox Approval Workflow (ADR-005)

The 7-stage remediation workflow provisions isolated ECS Fargate environments, runs comprehensive test suites, and gates production deployment behind human approval.

StageNameHITL Gate?Description
1Detection & GenerationAgent detects vulnerability, Coder generates patch
2Sandbox ProvisioningIsolated ECS/Fargate environment with network isolation
3Sandbox TestingUnit, integration, security, and performance tests
4HITL NotificationGATEApproval request created, reviewers notified (SNS/SES/Slack)
5Human DecisionGATEReviewer sees diff, test results, scan results → APPROVE / REJECT / MODIFY
6Production DeploymentCI/CD pipeline deploys approved patch
7Post-Deploy Verification24-hour monitoring for errors, perf regression, security events
What the Reviewer Sees Original vulnerable code, AI-generated patch diff, sandbox test results (pass/fail counts), SAST/SCA security scan results, deployment plan with blast radius estimate, and any Constitutional AI critique findings.

7. Constitutional AI Escalation (ADR-063)

The critique pipeline evaluates agent outputs against 16 constitutional principles. MAJOR and CRITICAL severity critiques automatically escalate to HITL.

Critique SeverityActionHITL?
MINORStyle issues — non-blockingNo
MODERATEPotential logic errors — loggedNo
MAJORSecurity concerns, anti-patternsYes — Escalate
CRITICALDangerous outputs, policy violationsYes — Immediate

11. Audit & Compliance

Every HITL decision produces an immutable audit record covering who, when, what, why, and result. Three DynamoDB tables with retention policies aligned to SOX, HIPAA, and PCI-DSS.
