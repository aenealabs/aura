# Neptune Taint Resolver -- Operations Runbook

**ADR:** ADR-093 Neptune-Backed Cross-File Taint Resolver
**Owner:** Vulnerability scanner platform team
**Last reviewed:** May 13, 2026
**Deploy posture:** Code-complete; live deployment Deferred (Cost Gate).
**RTO / RPO:** **RTO = 0** (degrade-to-in-memory is automatic) / **RPO = ∞** (cache-class data; do not restore from snapshot -- re-derive instead).

---

## 1. What this is

The Neptune-backed taint resolver persists per-function taint summaries (`returns_tainted`, `sink_params`) to Neptune at the end of each scan so future scans can read them back instead of re-deriving. It is the load-bearing piece of the incremental-scan story.

**Important framing:** persisted summaries are **a cache, not a system of record.** Every summary can be regenerated by re-parsing the source. There is no PITR on this cluster; backups are 1-day operational only (Neptune minimum). If summaries become corrupted, the operator runs the kill-switch and the next scan re-derives from source.

**Backends in play:**

| Component | Location | Purpose |
|---|---|---|
| Neptune cluster | `neptune.aura.local:8182` | Stores `VulnCodeUnit` vertices + `SINKS_PARAM` / `DEPENDS_ON_SUMMARY` edges |
| DynamoDB `aura-scan-state` | per-region | Per-scan `taint_write_complete` write-fence flag (Sally C-4) |
| DynamoDB `aura-summary-revocations` | per-region | Revocation entries consulted on every read (Sally C-2) |
| KMS per-tenant asymmetric CMK | alias pattern `alias/aura/scanner/taint-signing-<tenant>` | Signs every persisted summary (`RSASSA_PSS_SHA_256`) |
| SSM Parameter Store `/aura/scanner/neptune_taint_resolver_enabled` | per-region | Kill-switch flag polled at every scan start |
| CloudWatch namespace `Aura/Scanner/TaintResolver` | per-region | 5 named alarms (see §6) |

## 2. The 5 mandatory alarms

| Alarm | Threshold | Severity | First-response action |
|---|---|---|---|
| `summary.bad_signature` | `>0 in 5min` | **P1** | Possible tampering or revoked key in use. Run §4 kill-switch immediately; investigate per §7. |
| `summary.cache_miss_rate` | `>50% over 30min` | P2 | Preload underperforming or schema drift; check Neptune p99 + recent deploys. |
| `summary.write_fence_timeout` | `>0 in 5min` | P2 | Neptune write or DynamoDB fence failed; check both. Acceptable transient if cleared in <30min, else escalate. |
| `gremlin.p99_latency_ms` | `>500 over 15min` | P3 | Read path degrading; check Neptune writer-instance CPU + connection-pool exhaustion. |
| `tenant_predicate_rejection` | `>0 ever` | **P1** | Either a code bug or an active probe. **Capture the bytecode that was rejected**; engage security on-call before resolving. |

---

## 3. The <5-minute kill-switch (degrade to in-memory)

Use this when ANY of the following is true:

* `summary.bad_signature` alarm fires.
* `tenant_predicate_rejection` alarm fires.
* Suspected tenant-isolation breach (cross-tenant data in a scan's output).
* Signing-key compromise suspected.
* Neptune cluster is degraded (p99 > 5s sustained).

**Procedure:**

```bash
# 1. Disable the Neptune-backed resolver across the fleet.
./scripts/disable_neptune_taint_resolver.py \
    --region us-east-1 \
    --operator "$(whoami)"

# 2. Verify the flag flipped (CloudTrail will also show the change).
aws ssm get-parameter \
    --region us-east-1 \
    --name /aura/scanner/neptune_taint_resolver_enabled \
    --with-decryption \
    --query Parameter.Value --output text
# Expected: false
```

After the flip:
* **Every new scan** falls back to the in-memory `CrossFileTaintContext` automatically. No code restart required.
* **Already-running scans** complete on whichever backend they started with -- there is no mid-scan switch.
* The `taint.resolver.degraded_to_memory.flag_off` metric (under `Aura/Scanner/TaintResolver`) increments on the next scan, confirming the flag is being honored.

**To re-enable** after the incident is resolved:

```bash
./scripts/disable_neptune_taint_resolver.py \
    --region us-east-1 \
    --enable \
    --operator "$(whoami)"
```

## 4. Revoking compromised summary signatures

Use this when:

* A scanner worker pod is compromised between time `T1` and `T2`.
* A signing-bug release shipped between `T1` and `T2`.
* CloudTrail shows anomalous `kms:Sign` activity.

Every summary signed by `signer_arn` within `[T1, T2]` must be treated as untrustworthy. The resolver's `DynamoDbRevocationOracle` consults the `summary_revocations` table on every read; entries in this table cause the matching summaries to be dropped + re-derived on the next scan.

**Procedure (5 steps):**

```bash
# Step 1: Kill-switch (degrade everyone to in-memory while we work).
./scripts/disable_neptune_taint_resolver.py --region us-east-1

# Step 2: Write the revocation entry.
./scripts/revoke_summary_signatures.py \
    --region us-east-1 \
    --signer-arn arn:aws:kms:us-east-1:111122223333:key/<COMPROMISED_KEY> \
    --from 2026-05-13T14:00:00+00:00 \
    --to   2026-05-13T16:30:00+00:00 \
    --reason "worker-pod compromise; potential signature exfiltration"

# Step 3: Verify the entry was written.
aws dynamodb query \
    --region us-east-1 \
    --table-name aura-summary-revocations \
    --key-condition-expression 'signer_arn = :s' \
    --expression-attribute-values '{":s":{"S":"arn:aws:kms:..."}}'

# Step 4: Revoke the worker's IAM kms:Sign permission at IAM (out of scope
# for this script; coordinate with security on-call).

# Step 5: Re-enable the resolver. Revoked summaries will be detected on
# every preload and re-derived; the rest of the fleet returns to normal.
./scripts/disable_neptune_taint_resolver.py --region us-east-1 --enable
```

**Background quarantine job:** there is a separate process (not in scope for this runbook) that finds vertices in Neptune whose signature falls within a revoked window and marks them `quarantined=true`. The next scan deletes them. Engage the scanner platform team if you need to force this job.

## 5. Stale-summary suspicion (false positive in production)

Symptom: a known-good function is suddenly flagged with a `cross-file` taint chain that no longer reflects reality (e.g., a sanitizer was added but the chain still emits).

**Most likely cause:** a stale persisted summary whose `content_hash` no longer matches the current function body, OR a stale transitive `DEPENDS_ON_SUMMARY` edge.

**Procedure:**

```bash
# 1. Kill-switch to in-memory.
./scripts/disable_neptune_taint_resolver.py --region us-east-1

# 2. Wait for one full scan cycle of the affected repo (typically <30 min)
#    so the in-memory backend re-derives the chain set from source.

# 3. Verify the false positive disappears in the in-memory chains.

# 4. If it disappears in-memory but reappears with Neptune enabled, the
#    persisted summary is stale. Two options:
#    (a) Bump TAINT_SCHEMA_VERSION in code -> next scan re-derives.
#    (b) Revoke the signer for the time window of the stale summary.
#        See §4.

# 5. Re-enable.
./scripts/disable_neptune_taint_resolver.py --region us-east-1 --enable
```

## 6. Disaster recovery -- "Neptune is gone"

**RTO target: 0.** The resolver's whole-scan fail-safe degrades automatically; no operator action is required for continuity of service.

If the entire Neptune cluster is destroyed:
1. The factory's preload step fails -> degrades every new scan to `InMemoryTaintContext`.
2. The `taint.resolver.degraded_to_memory.preload_failed` metric will spike (alarm threshold pending Phase 5 tuning).
3. Operator rebuilds the Neptune cluster from CloudFormation; **no snapshot restore is needed** -- the cluster's content is cache-class. Future scans will re-derive and re-persist summaries.

**Do not** attempt a PITR restore. The cluster is provisioned with `BackupRetentionPeriod: 1` (AWS minimum) intentionally; the 1-day snapshot exists only for operational sanity, not for DR. Restoring stale summaries is a correctness risk; re-deriving is the correct DR path.

## 7. Suspected tampering / `summary.bad_signature` triage

This is the highest-severity alarm. Steps:

1. **Kill-switch immediately** (§3). Buy time before deciding what happened.
2. **Capture the failing summary's metadata.** CloudWatch Log Group `/aura/scanner/taint-summaries` records `(scan_id, tenant_id, signer_arn, summary_hash)` for every signature failure.
3. **Check KMS CloudTrail** for the named `signer_arn`. Look for: (a) unexpected `kms:Sign` callers, (b) `kms:Sign` at unusual times/volumes, (c) `kms:Disable*` events.
4. **If KMS shows clean activity:** the failure is likely transient (KMS throttling, race condition). Wait 15 min, check `summary.bad_signature` rate, re-enable if it's stayed at 0.
5. **If KMS shows compromise indicators:** treat as confirmed compromise. Execute §4 with the full compromise window. Engage security on-call. File an incident.
6. **After re-enabling**, watch `summary.bad_signature` for the next hour. Re-revoke if it re-fires.
