1. Overview

Hybrid GraphRAG is Aura's retrieval architecture that combines graph databases with vector search to provide comprehensive code understanding for autonomous vulnerability detection and remediation.

What is Hybrid GraphRAG? — Traditional RAG (Retrieval-Augmented Generation) uses vector similarity search alone. Aura's Hybrid GraphRAG adds structural code analysis through graph databases and keyword matching through BM25, enabling queries like "find all functions that call this vulnerable method" alongside semantic queries like "find similar authentication code."
Dense Retrieval (Vector k-NN)
Semantic similarity via embedding comparison using OpenSearch k-NN with 1536-dimensional vectors. Captures meaning and finds code pattern similarities across the codebase.
Sparse Retrieval (BM25)
Keyword matching via traditional text search with field-specific boosting: function_names (3x), file_path (2x), content (1x). Critical for exact identifier matches.
Graph Retrieval (Gremlin)
Structural relationship traversal via AWS Neptune property graph. Navigates call graphs, dependencies, inheritance, and references with up to 3-hop traversal.

2. Why Hybrid Retrieval Matters

Code has two types of relationships: structural (explicit, deterministic connections) and semantic (implicit, similarity-based). Neither alone provides complete context for security analysis.

Structural Relationships (Graph)

  • Call graphs — Function A calls Function B
  • Dependencies — Module X imports Module Y
  • Inheritance — Class Child extends Class Parent
  • References — Variable usage across files

Semantic Relationships (Vector)

  • Pattern similarity — Code that does similar things
  • Intent matching — Code that serves similar purposes
  • Contextual relevance — Code related to a concept
Retrieval TypeStrengthsLimitationsBest For
Structural (Graph)Precise relationships, deterministicMisses semantic similarityImpact analysis, dependency tracing
Semantic (Vector)Captures meaning, finds patternsMisses explicit connectionsFinding similar code, pattern matching
Keyword (BM25)Exact identifier matchesNo semantic understandingFinding specific function/variable names
Hybrid (All Three)Comprehensive contextRequires fusion logicComplete security analysis
Research-Validated: Aura's three-way hybrid approach achieves 22-25% improvement in mean reciprocal rank (MRR) compared to single-method approaches. Adding BM25 sparse search to dense vectors and graph traversal captures exact identifier matches that vector search alone misses.

4. Three-Way Retrieval & RRF Fusion

When a query is issued, all three retrieval methods execute concurrently and results are fused using Reciprocal Rank Fusion (RRF).

Retrieval Flow

Query: "Find code that validates user authentication tokens" Query Analysis | | Determine which retrieval methods are relevant | semantic_keywords: ["validates", "authentication", "tokens"] | graph_traversal: true vector_search: true bm25_search: true | +-------------------+-------------------+ | | | v v v Dense (k-NN) Sparse (BM25) Graph (Gremlin) OpenSearch OpenSearch Neptune 50 results 50 results 50 results | | | +-------------------+-------------------+ | v RRF Fusion (k=60) Score = SUM(weight / (k + rank)) | v Top 20 Results (deduplicated, enriched)

Reciprocal Rank Fusion (RRF)

Results from all sources are combined using RRF, which balances precision and recall without requiring score normalization across different retrieval systems:

// RRF formula RRF_Score = SUM( weight / (k + rank) ) // Where: // k = 60 (constant to dampen high-ranking outliers) // rank = position in source result list (1-indexed) // weight = source-specific weight

Default Source Weights

SourceWeightRationale
Dense (Vector k-NN)1.0Baseline semantic relevance
Sparse (BM25)1.2Boost for exact keyword matches (research-tuned)
Graph (Gremlin)1.0Structural relationships
Sparse Boost (1.2) is critical. BM25 excels at finding exact function names and identifiers that vector search might miss. This weight was empirically tuned from retrieval evaluation benchmarks.

6. Graph Query Patterns (5 Types)

Aura implements five structured graph query patterns for security-focused structural code analysis, defined in the GraphQueryType enum.

class GraphQueryType(Enum): CALL_GRAPH = "call_graph" # Functions that call / are called by target DEPENDENCIES = "dependencies" # Import relationships INHERITANCE = "inheritance" # Class hierarchies REFERENCES = "references" # All references to an entity RELATED = "related" # General related entities
CALL_GRAPH
Use case: Impact analysis for vulnerability fixes

Find all functions that call a specific vulnerable function, enabling precise blast-radius assessment before patching.
DEPENDENCIES
Use case: Dependency vulnerability assessment

Find all files that depend on a vulnerable package, revealing the full scope of exposure across the codebase.
INHERITANCE
Use case: Inherited vulnerability detection

Find all classes inheriting from an insecure base class, traversing the full hierarchy with recursive descent.
REFERENCES
Use case: Secret exposure analysis

Find all usages of a specific variable or constant (e.g., API_KEY), tracing references across functions and files.
RELATED
Use case: Comprehensive context gathering

Find structurally related code through multiple relationship types: callers, callees, and co-located functions.

Example Gremlin Queries

// CALL_GRAPH: Find all callers of a vulnerable function g.V().has('Function', 'name', 'insecure_hash') .in('CALLS') .project('caller', 'file', 'line') .by('name') .by(out('CONTAINED_IN').values('path')) .by('line_start') // DEPENDENCIES: Find files depending on a vulnerable package g.V().has('Package', 'name', 'vulnerable-lib') .in('DEPENDS_ON') .values('path') // INHERITANCE: Recursive class hierarchy traversal g.V().has('Class', 'name', 'InsecureBase') .in('EXTENDS') .emit() .repeat(in('EXTENDS')) .until(__.not(in('EXTENDS'))) // REFERENCES: Trace variable usage across codebase g.V().has('Variable', 'name', 'API_KEY') .in('REFERENCES') .project('function', 'file') .by('name') .by(out('CONTAINED_IN').values('path')) // RELATED: Multi-relationship context gathering g.V().has('Function', 'name', 'target_function') .union( in('CALLS'), // Functions that call this out('CALLS'), // Functions this calls out('CONTAINED_IN') // File containing this .in('CONTAINED_IN') // Other functions in same file ) .dedup()

7. Context Engineering Stack (6 Layers)

The Context Stack Manager implements explicit layered context management, ensuring optimal token allocation across different context types.

Layer 1: System Instructions (2K tokens)

Agent identity, behavioral rules, and operational constraints. Highest priority, always included.

Layer 2: Long-Term Memory (10K tokens)

Titan Neural Memory integration (ADR-024). Persistent learned patterns and cross-session knowledge.

Layer 3: Retrieved Documents (50K tokens)

Hybrid GraphRAG context — the primary output of the three-way retrieval system. Largest token allocation.

Layer 4: Tool Definitions (3K tokens)

Hierarchical tool registry with ~20 atomic tools organized in 3 tiers for efficient selection.

Layer 5: Conversation History (20K tokens)

Session context with community summarization for compression. Preserves dialogue continuity.

Layer 6: Current Task (15K tokens)

Immediate objective, active vulnerability analysis, or patch generation context.
Total Context Budget: ~100K tokens. Research finding: effective context window is <256K tokens. The 6-layer stack ensures optimal allocation with Layer 3 (Hybrid GraphRAG) receiving 50% of the budget for maximum retrieval depth.

8. HopRAG Multi-Hop Retrieval

HopRAG extends the base retrieval system with multi-hop graph traversal guided by semantic similarity, enabling complex reasoning chains across the codebase.

Pseudo-Query Edge Generation
LLM-powered generation of hypothetical query edges that link semantically related nodes even without explicit structural connections. Stored as RELATED_BY_QUERY edges in Neptune.
Retrieve-Reason-Prune Pipeline
Three-stage pipeline: (1) Retrieve candidate nodes, (2) Reason about relevance using LLM, (3) Prune low-confidence results. Pruning threshold: 0.5
Multi-Hop Traversal
Up to 3 hops through the graph with semantic guidance at each step. Each hop uses both structural edges and pseudo-query edges for navigation.
Embedding Cache
LRU cache of 1,000 items for frequently accessed embeddings. Reduces recomputation during multi-hop traversal and improves query latency.
HopRAG Performance (arXiv:2502.12442): +65.07% retrieval F1 improvement and +76.78% answer accuracy on complex multi-hop queries compared to single-hop retrieval.

9. Security & Integrity

GraphRAG context is a high-value attack surface. Aura implements multiple defense layers to protect against context poisoning and manipulation.

GraphRAG Security Service Security
807 lines of defense against context poisoning:
• Entity signature verification
• Anomaly detection on graph mutations
• Rate limiting on graph writes
• Audit logging for all modifications
Context Provenance (ADR-067) Core
Index-time provenance tracking with trust scoring:
• Every retrieved document includes provenance metadata
• Anomaly detection flags suspicious context
• Quarantine system for untrusted results
• 275 tests validating integrity
Input Sanitization (RLM) Security
Prompt injection prevention in retrieval inputs:
• RLM InputSanitizer validates all queries
• REPLSecurityGuard validates code in context
• 119 tests in the RLM package
MCP Context Isolation Core
Multi-agent context isolation via Model Context Protocol:
• Agents cannot access each other's context
• Prevents cross-contamination in multi-agent workflows
• Enforced at the protocol level

11. Configuration Options

Administrators can tune retrieval behavior through configuration to optimize for different codebase characteristics.

Retrieval Weights

// Adjust relative importance of each retrieval method { "retrieval_config": { "dense_weight": 1.0, // Baseline semantic relevance "sparse_boost": 1.2, // Boost for exact keyword matches "graph_weight": 1.0, // Structural relationships "rrf_k": 60 // RRF damping constant } }
Tuning guidance: Increase sparse_boost for codebases with unique naming conventions. Increase graph_weight for highly interconnected architectures. Decrease graph_weight for isolated microservices.

Index Configuration

{ "index_config": { "include_patterns": ["*.py", "*.js", "*.ts", "*.java"], "exclude_patterns": ["**/test/**", "**/vendor/**"], "chunk_size": 500, // tokens per chunk "chunk_overlap": 50, // overlap between chunks "embedding_model": "code-embedding-v2" } }
